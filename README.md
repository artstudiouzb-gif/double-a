# ArtStudio CMS

Лёгкая CMS на чистом PHP 8.2+ и MySQL/MariaDB без систем плагинов и тем.
Весь функционал зашит в ядро, дизайн страниц собирается из модульных
блоков (page builder) с изолированными (scoped) стилями.

Без Composer и внешних библиотек — только стандартные расширения PHP
(`pdo_mysql`, `session`, `hash`, `openssl`/`random_bytes`).

## Требования

- PHP 8.2+
- Расширения: `pdo_mysql`, `mbstring`, `json`
- MySQL 8.0+ или MariaDB 10.5+
- Apache с `mod_rewrite` (или эквивалент правил для nginx, см. ниже)

## Установка

1. **База данных.** Создайте базу и импортируйте схему:

   ```bash
   mysql -u root -p -e "CREATE DATABASE artstudio_cms CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci"
   mysql -u root -p artstudio_cms < database/schema.sql
   ```

2. **Конфигурация.** Скопируйте `config/config.example.php` в
   `config/config.php` и укажите реальные данные подключения к БД
   (или задайте переменные окружения `DB_HOST`, `DB_DATABASE`,
   `DB_USERNAME`, `DB_PASSWORD`, `APP_URL`, `APP_ENV=production` на
   стороне хостинга — тогда редактировать файл не нужно).

3. **Document root.** Направьте document root домена на папку
   `/public` — там лежит единственная точка входа `index.php` и
   `download.php`. Если хостинг не позволяет сменить document root
   (домен = корень аккаунта), в корне проекта уже лежит `.htaccess`,
   прозрачно проксирующий все запросы в `/public`.

4. **Права на папки.** Веб-серверу нужен доступ на запись в:
   - `storage/logs/`
   - `storage/protected_uploads/` (защищённые файлы, хранится вне `public/`)
   - `public/uploads/public/` (публичные файлы, отдаются напрямую)

5. **Первый администратор.** Аккаунтов по умолчанию нет — создайте
   первого администратора через консоль:

   ```bash
   php database/create_admin.php
   ```

   При первом входе в `/admin/login` система обязательно предложит
   подключить 2FA (TOTP) — без этого доступ к дашборду не выдаётся.

6. Откройте `https://ваш-домен/admin/login`.

## Nginx (если не Apache)

```nginx
root /path/to/artstudio/public;
index index.php;

location / {
    try_files $uri $uri/ /index.php?$query_string;
}

location ~ \.php$ {
    fastcgi_pass unix:/run/php/php8.2-fpm.sock;
    fastcgi_index index.php;
    include fastcgi_params;
    fastcgi_param SCRIPT_FILENAME $document_root$fastcgi_script_name;
}

location ~* ^/(storage|config|app|templates|database)/ {
    deny all;
}
```

## Безопасность входа в панель управления

- Пароли хранятся через `password_hash()` (bcrypt, cost 12).
- Обязательная 2FA (TOTP, RFC 6238) — реализована вручную в
  `App\Core\TOTP`, без внешних библиотек. Совместима с Google
  Authenticator, Яндекс Ключ и любым другим TOTP-приложением.
  Секретный ключ вводится вручную (без QR, чтобы не тянуть зависимости
  и не обращаться к сторонним сервисам генерации QR-кодов).
- Сессии: `HttpOnly`, `Secure` (авто-определение HTTPS), `SameSite=Lax`,
  `session.use_strict_mode`, регенерация ID при каждом повышении
  привилегий (после пароля, после 2FA).
- Rate limiting на вход и на проверку 2FA-кода: блокировка по паре
  `IP + логин` после `security.login_max_attempts` неудачных попыток
  в течение `security.login_attempts_window_minutes` минут
  (`app/Core/RateLimiter.php`, таблица `login_attempts`).
- CSRF-токен на всех POST-формах (`App\Core\Csrf`).

## Конструктор страниц и изоляция стилей блоков

Каждая страница (`pages`) хранит набор блоков (`blocks`, поле
`page_id`, `sort_order`). У блока есть `type`, JSON `data` и
произвольный `custom_css`.

При рендере (`App\Core\BlockRenderer::render()`):

1. HTML блока рендерится из шаблона `templates/blocks/{type}.php` и
   оборачивается в `<section id="block-{id}" class="cms-block ...">`.
2. `custom_css` блока прогоняется через `App\Core\CssScoper::scope()`,
   который разбирает CSS по фигурным скобкам (а не наивной регуляркой)
   и добавляет префикс `#block-{id}` к каждому селектору верхнего
   уровня, рекурсивно спускаясь внутрь `@media`/`@supports` и оставляя
   `@keyframes`/`@font-face` без изменений. Также вырезаются `@import`,
   `javascript:`, `expression()` — типовые векторы CSS-based атак.
3. Итоговый CSS каждого блока собирается в единый `<style>` в `<head>`
   страницы (`App\Core\BlockRenderer::renderPage()`), поэтому стили
   одного блока физически не могут задеть селекторы другого блока или
   общий дизайн сайта.

Готовые типы блоков: `text`, `html`, `cta`, `advantages`, `slider`,
`gallery`, `form`. Новый тип блока добавляется одним файлом в
`templates/blocks/`.

## Файловый менеджер

- **Публичные файлы** — `public/uploads/public/`, отдаются веб-сервером
  напрямую по ссылке вида `/uploads/public/<stored_name>`.
- **Защищённые файлы** — `storage/protected_uploads/`, физически лежат
  **вне** document root (`public/`), поэтому недоступны по прямой
  ссылке в принципе, независимо от конфигурации `.htaccess`. Отдаются
  только через `download.php?file_id=X`, который проверяет:
  1. авторизован ли пользователь в панели управления (активная сессия), либо
  2. передан ли верный `token`, совпадающий с `files.access_token`
     (`hash_equals`, защита от timing-атак).

  Путь к файлу дополнительно проверяется через `realpath()` +
  `str_starts_with()` относительно базовой директории — защита от
  path traversal, даже если `stored_name` когда-либо будет неконтролируемым.

## Что реализовано

### Этап 1 — ядро

1. Полная схема БД (`database/schema.sql`) — пользователи, новости,
   страницы, блоки, проекты (+ галерея + кастомные поля), команда,
   формы + заявки, настройки, файлы, `login_attempts`.
2. Структура проекта (см. выше).
3. Ядро безопасности: логин + обязательная 2FA (TOTP), rate limiting,
   CSRF, безопасные сессии.
4. Механизм рендера страниц из блоков с изоляцией CSS.
5. `download.php` — контроллер безопасной отдачи защищённых файлов.

### Этап 2 — админ-панель

- **Новости** (`/admin/news`) — CRUD, статус черновик/опубликовано, дата
  публикации, загрузка изображения или ссылка на него.
- **Страницы + конструктор блоков** (`/admin/pages`) — CRUD страниц,
  назначение главной страницы; на странице редактирования — список
  блоков с перемещением вверх/вниз, добавлением/удалением, и
  текстовый редактор полей под каждый тип блока (`text`, `html`, `cta`,
  `advantages`, `slider`, `gallery`, `form`), включая собственный CSS
  с наглядным примером изоляции прямо в подсказке под полем.
- **Проекты** (`/admin/projects`) — CRUD, галерея изображений
  (`project_images`) и произвольные кастомные поля (`project_fields`)
  через повторяющиеся группы полей (добавить/удалить строку).
- **Команда** (`/admin/team`) — CRUD, фото (файл или ссылка),
  соцсети (Facebook/Instagram/Telegram/LinkedIn/WhatsApp).
- **Формы** (`/admin/forms`) — конструктор полей (имя, подпись, тип,
  обязательность), список заявок с пометкой прочитанных и удалением.
  Приём заявок с сайта — `POST /forms/{slug}/submit`
  (`App\Controllers\Site\FormController`): валидация обязательных
  полей и email, CSRF, best-effort уведомление на email через `mail()`.
- **Настройки дизайна** (`/admin/settings`) — логотип, цвета, шрифт,
  контакты, коды счётчиков; применяются на сайте сразу через
  `App\Models\Setting`.
- **Файловый менеджер** (`/admin/files`) — загрузка (публично или
  защищённо) через `App\Core\Uploader` с проверкой реального
  MIME-типа содержимого (а не расширения из имени), список файлов со
  ссылками, перегенерация токена доступа, удаление (файл на диске +
  запись в БД).
- **Новости на сайте** — `/news` (список опубликованных) и
  `/news/{slug}` (страница новости).
- Общий layout админки (`app/Views/admin/layout/`) и общий layout
  сайта (`app/Views/site/_header.php` / `_footer.php`), flash-сообщения
  (`App\Core\Flash`) на обеих сторонах.
- Транслитерация кириллицы в ЧПУ-слаги (`App\Core\Slug`) для новостей,
  страниц, проектов, форм.

Все перечисленные сценарии стадии 2 проверены end-to-end на реальном
MariaDB + встроенном PHP-сервере: создание/редактирование сущностей,
добавление блоков и проверка их вывода со scoped CSS на главной
странице, отправка формы с фронтенда с валидацией и flash-сообщениями,
загрузка/скачивание/удаление публичных и защищённых файлов, отклонение
запрещённых типов файлов (например, `.php`).

### Этап 3 — мультиязычность, конструктор шапки, модульные виджеты

- **Мультиязычность** (`languages`, `page_translations`, `news_translations`,
  `blocks.lang`):
  - Управляемый список языков в админке (`/admin/languages`): код, название,
    язык по умолчанию, активность, порядок. Язык по умолчанию удалить нельзя;
    ровно один язык всегда остаётся дефолтным.
  - Язык по умолчанию — URL без префикса (`/news`), остальные активные —
    с префиксом (`/uz/news`). Определение языка в `App\Core\Router`
    (`resolveLocale`) + контекст `App\Core\Locale`.
  - Новости и страницы: базовая строка = язык по умолчанию, переводы в
    зависимых таблицах с составным ключом `(entity_id, lang)`. Редактирование
    всех языков на одной странице через вкладки (`data-lang-tabs`). Пустые
    поля перевода на сайте заменяются версией основного языка (graceful
    fallback).
  - У каждого языка страницы — независимый стек блоков (`blocks.lang`).
    Если стек языка пуст — на сайте показывается стек основного языка.
- **Конструктор шапки (mini-app)** (`/admin/header`, JSON в
  `settings.header_config`, `App\Core\HeaderConfig`): положение логотипа
  (left/center), положение меню (left/center/right), переключатель языков
  (вкл/выкл, формат: ISO-код / название / флаг-эмодзи), кнопки соцсетей
  (динамический список), CTA-кнопка (вкл/выкл, текст, ссылка, стиль
  залитая/контурная). `app/Views/site/_header.php` динамически
  перестраивает сетку шапки по трём зонам без хардкода.
- **Конструктор меню** (`/admin/menu`, `menu_items`): пункты с типом
  ссылки (страница по slug / раздел новостей / произвольный URL),
  привязкой к языку или ко всем языкам, сортировкой. URL пунктов
  автоматически получают языковой префикс.
- **Модульные боковые виджеты (Sidebar Engine)** (`widgets`,
  `pages.layout_type`, `templates/widgets/*.php`, `App\Core\WidgetRenderer`):
  - Макет страницы: `no_sidebar` / `left_sidebar` / `right_sidebar`.
  - Виджеты подключаются модульно, как блоки: шаблон каждого типа в
    `templates/widgets/{type}.php`, оборачивается в `<aside id="widget-{id}">`.
  - Типы: последние новости, контакты + соцсети, произвольный HTML/баннер,
    список проектов, список команды.
  - Менеджер (`/admin/widgets`): назначение в левую/правую колонку,
    порядок, язык (конкретный или все), JSON-настройки, вкл/выкл.
  - JSON-данные виджета сливаются с дефолтной структурой при рендере —
    старые/неполные данные не вызывают ошибок.

Этап 3 проверен end-to-end на MariaDB: управление языками (добавление EN
через админку), мультиязычные страница и новость с переводами и раздельными
стеками блоков, вывод сайта на `/` (ru) и `/uz` с корректным контентом,
scoped CSS, локализованными ссылками и переключателем языков; конструктор
шапки (логотип/меню/переключатель-флаги/соцсеть/CTA); конструктор меню с
префиксацией URL; левый и правый сайдбары с виджетами; fallback стека блоков
на язык по умолчанию; миграция `database/migrations/` на схему этапов 1–2
(включая backfill `blocks.lang`); отсутствие регрессий в разделах этапов 1–2.

Для инсталляций, развёрнутых на схеме этапов 1–2, применить миграцию:


### Этап 4 — безопасность и отказоустойчивость (Блок 1)

- **SVG-санитайзер** (`App\Core\Uploader::sanitizeSvgString`): загружаемые
  SVG очищаются через `DOMDocument` (нативно, без зависимостей) — удаляются
  `<script>`, `<foreignObject>`, обработчики `on*`, `javascript:`/`vbscript:`
  в `href`/`src`/`style`. Невалидный XML заменяется безопасной заглушкой
  (fail-closed). Дополнительно `download.php` отдаёт SVG как `attachment` с
  `Content-Security-Policy: default-src 'none'; sandbox`.
- **Привязка сессий** (`App\Core\Auth`): при входе сохраняется фингерпринт
  `sha256(User-Agent + подсеть /16)`; `Auth::check()` сверяет его на каждом
  запросе и уничтожает сессию при несовпадении (защита от перехвата cookie).
- **Нативный QR-код для TOTP** (`App\Core\QrCode`): генератор QR на чистом
  PHP (byte-режим, Reed–Solomon над GF(256), 8 масок с выбором по штрафным
  очкам, версии 1–6, уровень M) → SVG. otpauth-URI компактный (без
  algorithm/digits/period — это дефолты RFC 6238), укладывается в версию ≤6.
  На странице настройки 2FA показывается QR **и** ключ для ручного ввода
  (fallback). Корректность проверена сканированием сгенерированных кодов
  реальным QR-декодером (zbar).
- **Глобальный обработчик ошибок** (`App\Core\ErrorHandler`, подключён в
  `bootstrap.php`): перехватывает ошибки, исключения и фатальные ошибки. В
  production логирует полный стек в `storage/logs/error.log` и отдаёт
  `errors/500.php` с HTTP 500 без раскрытия трейса; в режиме отладки
  показывает детали.
- **Garbage collector и ротация логов** (`App\Core\Logger`,
  `RateLimiter::garbageCollect`): вероятностно при успешном входе удаляются
  записи `login_attempts` старше суток и ротируются лог-файлы по размеру
  (5 МБ, до 5 архивов `.log.1..5`), чтобы логи не разрастались.

Этап 4 проверен end-to-end: санитизация загруженного вредоносного SVG
(script/onload вырезаны, фигуры сохранены); инвалидация сессии при смене
User-Agent; QR со страницы 2FA декодируется реальным сканером в исходный
otpauth-URI; 500-страница в production с логированием трейса; очистка старых
login_attempts и ротация переросшего лога.

### Этап 5 — инфраструктура и управление данными (Блок 4)

- **Нативный SMTP-клиент** (`App\Core\Mailer`): чистый PHP через
  `stream_socket_client`, без `mail()` и внешних библиотек. Поддерживает
  STARTTLS (587), implicit SSL (465), `AUTH LOGIN`; корректный MIME с UTF-8
  (base64 темы/тела). Настраивается блоком `mail` в `config.php`
  (host/port/encryption/username/password/from). Уведомления форм переведены
  с `mail()` на `Mailer`; если SMTP не настроен — уведомление пропускается,
  заявка всё равно сохраняется.
- **CLI-миграции** (`database/migrate.php` + таблица `migrations`): сканирует
  `database/migrations/*.sql` по алфавиту имён, накатывает новые, фиксирует
  применённые. `php database/migrate.php status` — статус. Полный `schema.sql`
  помечает существующие миграции как применённые, поэтому свежая установка их
  не повторяет, а старые установки этапов 1–2 догоняют схему.
- **Soft deletes / корзина** (`deleted_at` у `pages`, `news`, `projects`):
  удаление помечает запись как удалённую; публичные и админ-списки фильтруют
  `deleted_at IS NULL`. Раздел `/admin/trash` показывает удалённое с
  восстановлением и окончательным удалением.
- **One-click backup** (`App\Core\Backup`): портируемый дамп БД через PDO
  (`SHOW CREATE TABLE` + `INSERT`, без внешнего `mysqldump`) плюс папки
  загрузок — всё в один `.zip` (нативный `ZipArchive`). CLI:
  `php database/backup.php`; в админке — кнопка «Скачать бэкап» в настройках
  (архив стримится и удаляется с диска). Дамп восстанавливается импортом в
  чистую БД.

Этап 5 проверен end-to-end: доставка письма через локальный SMTP по AUTH
LOGIN и по STARTTLS (aiosmtpd); накат миграции на схему этапов 1–2 и
идемпотентность; soft-delete → корзина → restore/force-delete; создание
бэкапа (CLI и через админку) и успешное восстановление дампа в новую БД.

### Этап 6 — веб-инсталлятор (Блок 8) + fail-safe

- **Установка в один клик**: если нет `config/config.php` или
  `storage/installed.lock`, `bootstrap.php` переводит весь трафик на
  `App\Controllers\InstallController` (мастер из 4 шагов):
  1. **Проверка окружения** (`App\Core\EnvironmentCheck`): PHP 8.2+,
     расширения `pdo_mysql`/`mbstring`/`json`/`gd`, права на запись
     `config/`, `storage/`, `public/uploads/`.
  2. **База данных**: проверка подключения к MySQL, создание БД (если её
     нет), импорт `database/schema.sql`, генерация `config/config.php`.
  3. **Сайт и локаль**: название, описание, часовой пояс, язык по умолчанию.
  4. **Супер-администратор**: логин/email/пароль (обязательная 2FA
     настраивается при первом входе — уже реализованный поток).
- **Installed lock**: по завершении создаётся `storage/installed.lock`; при
  его наличии любой доступ к установщику возвращает **403** (`guardInstalled`).
- **Fail-safe для БД (задача 45)**: `Database::init` бросает исключение
  вместо `exit`; в рабочем режиме недоступность БД отдаёт брендированный
  **503** (`errors/503.php`) без утечки трейса, в режиме установки —
  подключается мягко.

Этап 6 проверен end-to-end в чистом окружении (без `config.php`): редирект
на установщик, все 4 шага (создание БД + импорт схемы + генерация конфига,
настройка сайта со сменой языка по умолчанию и таймзоны, создание админа),
создание `installed.lock`, блокировка установщика (403) после установки,
работоспособность сайта и вход с обязательной 2FA, а также брендированный
503 при недоступной БД.

## Установка

Разверните файлы на хостинге, направьте document root на `/public`, откройте
сайт в браузере — откроется мастер установки. Либо используйте ручной путь:
импорт `database/schema.sql` + копирование `config/config.example.php` в
`config/config.php` + `php database/create_admin.php` (см. выше).

### Этап 7 — производительность, ассеты и SEO (Блок 2)

- **SEO-метаданные**: `site/_header.php` выводит `canonical`, Open Graph
  (`og:title`/`og:description`/`og:image`/`og:url`/`og:type`/`og:site_name`)
  и `twitter:card`. Новости отдают `og:type=article` и `og:image` из своего
  изображения; страницы — с фолбэком на логотип. Абсолютные URL строятся из
  `app.url` и текущей локали.
- **sitemap.xml и robots.txt** (`App\Core\Site\SitemapController`):
  динамический `/sitemap.xml` со всеми опубликованными страницами и новостями
  на каждом активном языке (с `lastmod`/`priority`) и `/robots.txt` со
  ссылкой на карту и запретом `/admin`, `/install`, `/download.php`.
- **AssetCollector** (`App\Core\AssetCollector`): JS-зависимости блоков
  (`slider.js`, `gallery.js`) подключаются внизу страницы строго один раз,
  даже если одинаковых блоков несколько (проверено: 3 слайдера → один
  `slider.js`). Блочные скрипты вынесены в `public/assets/js/blocks/`.
- **Файловый кэш** (`App\Core\Cache`): скомпилированные блоки страницы
  (HTML + scoped CSS + список ассетов) кэшируются в `storage/cache/` по ключу
  `page:{id}:{lang}` и сбрасываются при изменении страницы/блоков в админке
  (`Cache::forgetPrefix`). Ассеты корректно регистрируются и при попадании в
  кэш.
- **GD-оптимизация изображений** (`App\Core\Uploader::optimizeImage`): при
  загрузке JPEG/PNG нативным GD создаются WebP-версия полного размера и
  адаптивные разрешения `-1600.webp` (desktop) и `-800.webp` (mobile).
  Best-effort — ошибки не мешают загрузке.

Этап 7 проверен end-to-end: og-теги и canonical на страницах, валидные
`robots.txt`/`sitemap.xml`; дедупликация `slider.js` при трёх слайдерах; кэш
создаётся/отдаётся/инвалидируется при правке блока/репопулируется; загрузка
JPEG 2000×1200 порождает валидные WebP 2000/1600/800 px.

### Этап 8 — типографика, роли и жизненный цикл данных (Блок 7)

- **Мердж JSON блоков с дефолтами** (`App\Core\BlockRenderer::DEFAULTS`):
  сохранённые данные блока сливаются с дефолтной структурой по типу, поэтому
  неполные/старые JSON не приводят к обращению к несуществующим ключам.
  Дефолты — единый источник истины и для конструктора, и для рендера.
- **Микро-типограф и санитайзер** (`App\Core\TextProcessor`): при сохранении
  текстовых блоков чистит «мусор» из Word/Google Docs (условные комментарии,
  `<o:p>`, `mso-*` стили, `MsoNormal` и т.п.), расставляет неразрывные
  пробелы после коротких предлогов/союзов, конвертирует прямые кавычки в
  «ёлочки» (или „" для en) и дефисы в длинное тире. Для HTML обрабатываются
  только текстовые узлы (`<pre>`/`<code>` не трогаются).
- **Двухуровневый ACL** (`App\Core\Auth`): роли `super_admin` (исторически
  `admin`) и `editor`. Редактор управляет только контентом
  (новости, страницы, проекты, команда, формы, файлы, корзина); системные
  разделы (настройки, шапка, языки, меню, виджеты, пользователи) скрыты в
  навигации и защищены `requireSuperAdmin` (403). Поле кастомного CSS блока
  недоступно редактору (в т.ч. на уровне сервера). Управление учётными
  записями (`/admin/users`) — только для супер-администратора.
- **Каскадная очистка файлов-сирот** (`App\Core\MediaCleaner`): при
  окончательном удалении страницы/новости из корзины собираются привязанные
  медиафайлы, и те, что больше нигде не используются (в блоках, новостях,
  проектах, команде, настройках), удаляются с диска вместе со своими
  WebP-вариантами. Проверка путей — с защитой от path traversal.

Этап 8 проверен end-to-end: типографирование при сохранении блока (ёлочки,
nbsp, тире, вычистка Word-мусора); рендер блока с неполными данными без
PHP-warning'ов; editor получает 403 на системных разделах и в POST, супер —
полный доступ; окончательное удаление страницы удаляет её файл-сироту и
WebP-варианты, но сохраняет файл, используемый другой страницей.
